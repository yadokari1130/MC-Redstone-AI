description: レッドストーン回路の生成・検証

以下の手順に従って、ユーザーのリクエストに基づいたレッドストーン回路を Minecraft 内で生成し、検証してください。
全ての `mc-cli` コマンドは、実行可能なバイナリへのパスがすでに通っているため、**そのまま実行してください。バイナリへのパスを使ったり、poetryなどを使って実行を試みないでください。**
よい例：`mc-cli test xor_20260101_120000.yaml`
悪い例：`./mc-cli/mc-cli test xor_20260101_120000.yaml`、`poetry run python3 mc-cli.py test xor_20260101_120000.yaml`
基本的にはSkillsを参照しますが、必要に応じて`search_web`ツールを使ってWebから情報を取得してください。この時、英語のほうが信頼できる情報が多いため、英語で検索してください。
ただし、**`mc-cli`コマンドを使用する際は必ずSkillsを参照してください。**特に引数が複雑であるため、無駄なコマンド実行を防ぐために、必ず引数のフォーマットを確認してから実行してください。
**思考するときはすべて英語を使い、ユーザーとのやり取りにのみ日本語を使ってください。**

### 制約事項 (重要)
- **アクセス範囲の制限**: 閲覧・アクセスして良いディレクトリは `.agents/` および `temp/` のみに限定してください。ただし、知見まとめを書く際は `knowledges/` への書き込みも許可されます。
- **コマンドヘルプ**: `mc-cli`コマンドに関しては、`.agents/skills/mc_cli/`内の`SKILL.md`を確認してください。
- **ブロック情報**: 回路設計の際は、まず `.agents/skills/blocks/BLOCKS_SUMMARY.md` を参照してください。
- **向き（facing）の注意**: `repeater`, `comparator`, `observer` は向き（facing）の定義が直感的ではなく、配置を間違えやすいブロックです。これらを配置する際は、**必ず個別の Skills を参照して正しい向きの定義を確認してください。**
- **出力形式**: `mc-cli` のデフォルト出力はトークン削減のためコンパクトな配列形式 `[ID, [X, Y, Z], {Properties}]` になっています。
- **書き込み制限**: `temp/` および `knowledges/` ディレクトリ以下にファイルを生成する場合を除き、**絶対にファイルシステムへの書き込みを行わないでください。**
- **異常時の対応**: `mc-cli` の挙動がバグ（期待しないエラーや異常な動作）を示した場合は、**直ちに回路の生成を中断**し、ユーザーにバグの可能性を報告してください。
- **最大試行回数**: 手順7（テスト→修正サイクル）の試行回数は **最大5回** までとします。それを超えても成功しない場合は諦めてユーザーに報告してください。

## サブエージェントを使った実行フロー

このワークフローは **3つのフェーズ** に分かれており、各フェーズをサブエージェントに委譲して実行します。
各フェーズのサブエージェントは独立した実行単位であり、**前のフェーズが完了してから次のフェーズを開始**してください。
サブエージェントへの指示は、このワークフロー全体のコンテキスト（制約事項・ユーザーの要望・前フェーズの結果）をすべて含めて渡してください。
また、サブエージェントを呼び出す際は、**現在の親エージェントと同じモデル（例: Gemini 3.1 Pro）を指定して使用**するようにしてください。

---

### フェーズ1: 分析・計画サブエージェント（手順1〜3）

**このフェーズで実施すること**:
- ユーザーの要望確認（手順1）
- 周辺ブロック情報の取得（手順2）
- 実装計画の作成（手順3）

**サブエージェントへの指示に含めるべき情報**:
- ユーザーが要求した回路の種類・目的
- 設置する中心座標
- このワークフロー全体の制約事項

**サブエージェントが返すべき成果物**:
- 取得した周辺ブロック情報のサマリー（地面の高さ・障害物の有無など）
- 実装計画（使用するブロック・論理構造・具体的な配置方針）
- 既存ブロックが存在する場合の確認事項（ユーザーへの確認が必要な場合はここで中断して報告）

> **重要**: サブエージェントが実装計画を返したら、内容を確認し、設置予定場所に既存ブロックがある場合は **必ずユーザーに確認を取ってから**フェーズ2に進んでください。

---

### フェーズ2: 生成・準備サブエージェント（手順4〜6）

**このフェーズで実施すること**:
- 土台ブロックの設置（手順4）
- 回路 JSON の生成と保存（手順5）
- テスト YAML ファイルの生成（手順6）

**サブエージェントへの指示に含めるべき情報**:
- フェーズ1で得られた実装計画（ブロック配置方針・座標情報）
- `time`コマンドを使用した現在時刻の把握
- 保存先ファイル名の指定（命名規則: `temp/{name}_yyyymmdd_hhmmss.json` / `temp/{name}_test_yyyymmdd_hhmmss.yaml`）
- このワークフロー全体の制約事項

**サブエージェントが返すべき成果物**:
- 土台設置の完了報告（使用したブロックIDと設置範囲）
- 生成した回路 JSON ファイルのパス
- 生成したテスト YAML ファイルのパスと、テストケースの概要（真理値表）

> **重要**: サブエージェントが返したファイルパスを次のフェーズに引き継いでください。

---

### フェーズ3: テスト・修正サイクルサブエージェント（手順7）

**このフェーズで実施すること**:
- 回路 JSON を使ったブロック設置（手順7.1）
- テスト YAML の実行（手順7.1）
- テスト結果の評価と修正（手順7.2〜7.3）
- 最大5回までのリトライ（手順7.4）

**サブエージェントへの指示に含めるべき情報**:
- フェーズ2で生成した回路 JSON ファイルのパス
- フェーズ2で生成したテスト YAML ファイルのパス
- フェーズ1で作成した実装計画（修正時の参考情報として）
- このワークフロー全体の制約事項（最大試行回数5回など）

**サブエージェントが返すべき成果物**:
- 最終的なテスト結果（成功 / 失敗）
- 試行回数と、各試行で行った修正内容のサマリー
- 失敗した場合は、各試行における失敗の内容・原因の考察・解決できなかった問題点

> **重要**: サブエージェントが返した結果を受け取ったら、成功・失敗に関わらず手順8・手順9を実行してください。

### 1. ユーザーの要望分析
- ユーザーから回路の目的（例：「ANDゲートを作って」、「クロック回路を作って」など）と、設置する中心座標を受け取ってください。
- もし設置場所や入出力の方向など、回路の詳細構成に不足している情報があれば、作業を開始する前にユーザーに確認してください。

### 2. 周辺状況の取得
- `minecraft_get_blocks` スキルを使用して、指示された座標周辺（回路を設置する予定の範囲を含む十分な広さ）のブロック情報を取得してください。
- 取得した情報を分析し、地形（地面の高さ）や障害物、既存の構造物を確認してください。

### 3. 実装計画の作成
- 回路の論理構造、使用するブロック（レッドストーン、リピーター、コンパレータなど）、および具体的な配置図を含む実装計画を作成してください。
- **重要**: 手順2で取得した情報に基づき、設置予定場所に既にブロック（空気以外のブロック）が存在する場合は、**上書きして良いか、あるいは場所をずらすべきか**をユーザーに確認してください。

### 4. 土台の設置
- 回路を安定して動作させるため、`minecraft_fill_blocks` スキルを使用して土台となるブロックを配置してください。
- 今後の修正や拡張で回路がはみ出さないよう、少し広めに土台を作成することをお勧めします。
- `BLOCKS_SUMMARY.md` を参照して適切な土台ブロックとコンポーネントを選択してください。

### 5. 回路情報の生成と JSON 保存
- 設計した回路を構成するブロック情報の JSON 配列（`minecraft_place_blocks` で使用可能な形式）を作成してください。
- 生成した JSON は、`redstone_ai/temp/` ディレクトリ内にファイルとして保存してください。
  - **注意**: `redstone_ai/temp/` ディレクトリが存在しない場合は、まずディレクトリを作成してください。
- **ファイル命名規則**: `{name}_yyyymmdd_hhmmss.json` の形式でファイル名を決定してください。
  - `{name}` は回路の種類を表す英小文字のスネークケース（例: `xor_gate`, `clock_circuit`）
  - `yyyymmdd_hhmmss` は作業開始時の現在日時（例: `20260418_221800`）
  - 例: `temp/xor_gate_20260418_221800.json`
- **重要**: 既存のファイルを参照したり上書きしたりしないでください。必ず新しいファイル名で保存してください。
- 回路の修正を行う際も、同じファイル名で上書き保存してください（同一試行サイクル内のみ）。

### 6. テストファイルの生成
- `minecraft_test` スキルを参照して、テスト YAML ファイルを作成してください。
- **ファイル命名規則**: `{name}_test_yyyymmdd_hhmmss.yaml` の形式でファイル名を決定してください。
  - `{name}` と `yyyymmdd_hhmmss` は手順5のファイルと同じ値を使用してください。
  - 例: `temp/xor_gate_test_20260418_221800.yaml`
- **重要**: 既存のファイルを参照したり上書きしたりしないでください。必ず新しいファイル名で保存してください。
- テスト内容は、回路の真理値表・期待される動作に基づいて設計してください。
  - `setup` には手順5で生成した JSON ファイルを指定してください（`blocks_file` を使用）。
  - `steps` には入力操作（レバー・ボタンのインタラクト、`wait` など）を記述してください。
  - `assert` には出力ブロック（レッドストーンランプなど）の期待状態を記述してください。

### 7. テスト実行・修正サイクル（最大5回）

以下のサイクルを、成功するか最大5回に達するまで繰り返してください。

#### 7.1. 回路の設置とテスト実行
- `minecraft_place_blocks` スキルを使用し、手順5で作成した JSON ファイル（`@redstone_ai/temp/filename.json`）を引数に渡して、実際にブロックを設置してください。
- `minecraft_test` スキルを使用し、手順6で作成したテスト YAML ファイルを実行してください。
- **重要**: `mc-cli` の実行中に期待しないエラーや異常な挙動（例：座標のずれ、コマンドの失敗など）が発生した場合は、**その時点で作業を中断し、直ちにユーザーへバグの可能性を報告**してください。

#### 7.2. テスト結果の評価
- **全テスト合格の場合**: 手順8（完了報告）に進んでください。全テスト合格の条件は`mc-cli test`コマンドの終了ステータスが0のことを指します。それ以外の終了ステータスの場合は失敗とみなしてください。
- **1件でも失敗した場合**: 以下の追加分析ステップを実施してから、修正手順（7.3）に進んでください。

#### 7.2.1. 追加情報の収集（失敗時）

テスト結果の ON/OFF 情報だけでは原因が特定できない場合、以下の手段を活用して状況を詳しく分析してください。

- **ブロック状態の確認** (`minecraft_get_blocks` スキル):
  - 失敗したアサーション付近のブロックを `verbose` モードで取得し、プロパティ（`powered`, `lit`, `delay` など）や信号強度を確認してください。
  - コンパレーターやリピーターなど、状態が動作に影響するブロックは特に重点的に確認してください。

- **手動インタラクトによる動作確認** (`minecraft_interact_block` スキル):
  - 特定の入力状態でのブロックの挙動を確認したい場合、レバーやボタンを手動でインタラクトして中間状態を再現してください。
  - インタラクト後に再度 `minecraft_get_blocks` で周辺ブロックの状態を取得し、信号の流れを追跡してください。

収集した情報を分析し、原因を特定した上で次の修正手順（7.3）に進んでください。

#### 7.3. 修正手順（失敗時）
- 失敗の原因を分析し、回路 JSON（`temp/` 内のファイル）を修正してください。
  - まず元の JSON の設計を見直し、必要な箇所のみを修正してください。
  - とりあえず全体をリセットすることは避け、基本的には今ある回路の問題箇所を特定して修正してください。
  - 回路全体の設計が根本的に誤っている場合など、本当にすべてをリセットする必要がある場合のみリセットしてください。
- 修正後、7.1 に戻って再試行してください。

#### 7.4. 最大試行回数に達した場合
- 5回試行してもすべてのテストが合格しなかった場合は、**作業を中断**してください。
- ユーザーに以下の内容を報告してください：
  - 回路の生成に失敗したこと
  - 各試行で発生した失敗の内容と原因の考察
  - 考えられる根本的な問題点や解決のヒント
- その後、手順9（知見まとめ）に進んでください。

### 8. 完了報告
- 全テストが合格した場合、ユーザーに成功を報告してください。
- 報告内容には以下を含めてください：
  - 成功したこと・作成した回路の概要
  - 試行した回数と、途中で修正した内容（あれば）
  - 設置した座標範囲
- その後、手順9（知見まとめ）に進んでください。

### 9. 知見のまとめ（必須）

成功・失敗に関わらず、今回の回路生成で得た知見を `knowledges/` ディレクトリにマークダウンファイルとして保存してください。

- **保存先**: `redstone_ai/knowledges/` ディレクトリ（存在しない場合は作成してください）
- **ファイル命名規則**: `{name}_knowledge_yyyymmdd_hhmmss.md` の形式でファイル名を決定してください。
  - `{name}` と `yyyymmdd_hhmmss` は手順5・6のファイルと同じ値を使用してください。
  - 例: `knowledges/xor_gate_knowledge_20260418_221800.md`
- **重要**: 既存のファイルを参照したり上書きしたりしないでください。必ず新しいファイル名で保存してください。
- **内容**: 以下の項目を含めてください：

```markdown
# [回路名] 生成知見まとめ

## 概要
- 目的・作成した回路の説明
- 結果（成功 / 失敗）
- 試行回数

## うまくいったこと
- 成功した設計上の工夫・アプローチ
- 有効だったブロックの組み合わせや配置

## うまくいかなかったこと
- 失敗した試みと、その具体的な症状
- テスト結果（失敗したアサーション、期待値と実際値）

## 原因の考察
- 失敗の根本的な原因と推測
- 修正によって解決した内容（成功時）
- 解決できなかった問題点と今後の課題（失敗時）

## 学んだこと・次回への改善点
- 今回の経験から得た新しい知識や気づき
- 同様の回路を作る際に役立つポイント
```
