---
description: レッドストーン回路の生成・検証
---

以下の手順に従って、ユーザーのリクエストに基づいたレッドストーン回路を Minecraft 内で生成し、検証してください。
全ての `mc-cli` コマンドは、プロジェクトルートである `redstone_ai` ディレクトリで実行してください。
基本的にはSkillsを参照しますが、必要に応じて`search_web`ツールを使ってWebから情報を取得してください。
この時、英語のほうが信頼できる情報が多いため、英語で検索してください。

### 制約事項 (重要)
- **アクセス範囲の制限**: 閲覧・アクセスして良いディレクトリは `.agents/` および `temp/` のみに限定してください。ただし、知見まとめを書く際は `knowledges/` への書き込みも許可されます。
- **コマンド制限**: **使用してよいコマンドは`mc-cli`のみです。**そのほかのコマンドを使用する際は必ずユーザーに確認してください。
- **コマンドヘルプ**: `mc-cli`コマンドに関しては、`.agents/skills/mc_cli/`内の`SKILL.md`を確認してください。
- **ブロック情報**: 回路設計の際は、まず `.agents/skills/blocks/BLOCKS_SUMMARY.md` を参照してください。
- **出力形式**: `mc-cli` のデフォルト出力はトークン削減のためコンパクトな配列形式 `[ID, [X, Y, Z], {Properties}]` になっています。
- **書き込み制限**: `temp/` および `knowledges/` ディレクトリ以下にファイルを生成する場合を除き、**絶対にファイルシステムへの書き込みを行わないでください。**
- **異常時の対応**: `mc-cli` の挙動がバグ（期待しないエラーや異常な動作）を示した場合は、**直ちに回路の生成を中断**し、ユーザーにバグの可能性を報告してください。
- **最大試行回数**: 手順7（テスト→修正サイクル）の試行回数は **最大5回** までとします。それを超えても成功しない場合は諦めてユーザーに報告してください。

### 1. ユーザーの要望分析
- ユーザーから回路の目的（例：「ANDゲートを作って」、「クロック回路を作って」など）と、設置する中心座標を受け取ってください。
- もし設置場所や入出力の方向など、回路の詳細構成に不足している情報があれば、作業を開始する前にユーザーに確認してください。

### 2. 周辺状況の取得
- `minecraft_get_blocks` スキルを使用して、指示された座標周辺（回路を設置する予定の範囲を含む十分な広さ）のブロック情報を取得してください。
- 取得した情報を分析し、地形（地面の高さ）や障害物、既存の構造物を確認してください。

### 3. 実装計画の作成
- 回路の論理構造、使用するブロック（レッドストーン、リピーター、コンパレータなど）、および具体的な配置図を含む実装計画を作成してください。
- **重要**: 手順2で取得した情報に基づき、設置予定場所に既にブロック（空気以外のブロック）が存在する場合は、**上書きして良いか、あるいは場所をずらすべきか**をユーザーに確認してください。

### 4. 土台の設置
- 回路を安定して動作させるため、`minecraft_fill_blocks` スキルを使用して土台となるブロックを配置してください。
- 今後の修正や拡張で回路がはみ出さないよう、少し広めに土台を作成することをお勧めします。
- `BLOCKS_SUMMARY.md` を参照して適切な土台ブロックとコンポーネントを選択してください。

### 5. 回路情報の生成と JSON 保存
- 設計した回路を構成するブロック情報の JSON 配列（`minecraft_place_blocks` で使用可能な形式）を作成してください。
- 生成した JSON は、`redstone_ai/temp/` ディレクトリ内にファイルとして保存してください。
  - **注意**: `redstone_ai/temp/` ディレクトリが存在しない場合は、まずディレクトリを作成してください。
- 回路の修正を行う際も、この手順で JSON ファイルを更新してください。

### 6. テストファイルの生成
- `minecraft_test` スキルを参照して、テスト YAML ファイルを作成してください。
- テストファイルは `redstone_ai/temp/` ディレクトリ内に保存してください（例: `temp/circuit_test.yaml`）。
- テスト内容は、回路の真理値表・期待される動作に基づいて設計してください。
  - `setup` には手順5で生成した JSON ファイルを指定してください（`blocks_file` を使用）。
  - `steps` には入力操作（レバー・ボタンのインタラクト、`wait` など）を記述してください。
  - `assert` には出力ブロック（レッドストーンランプなど）の期待状態を記述してください。

### 7. テスト実行・修正サイクル（最大5回）

以下のサイクルを、成功するか最大5回に達するまで繰り返してください。

#### 7.1. 回路の設置とテスト実行
- `minecraft_place_blocks` スキルを使用し、手順5で作成した JSON ファイル（`@redstone_ai/temp/filename.json`）を引数に渡して、実際にブロックを設置してください。
- `minecraft_test` スキルを使用し、手順6で作成したテスト YAML ファイルを実行してください。
- **重要**: `mc-cli` の実行中に期待しないエラーや異常な挙動（例：座標のずれ、コマンドの失敗など）が発生した場合は、**その時点で作業を中断し、直ちにユーザーへバグの可能性を報告**してください。

#### 7.2. テスト結果の評価
- **全テスト合格の場合**: 手順8（完了報告）に進んでください。全テスト合格の条件は`mc-cli test`コマンドの終了ステータスが0のことを指します。それ以外の終了ステータスの場合は失敗とみなしてください。
- **1件でも失敗した場合**: 失敗したテストケース・アサーション内容・実際の状態を分析し、原因を特定してください。その後、以下の修正手順に従って再試行してください。

#### 7.3. 修正手順（失敗時）
- 失敗の原因を分析し、回路 JSON（`temp/` 内のファイル）を修正してください。
  - まず元の JSON の設計を見直し、必要な箇所のみを修正してください。
  - とりあえず全体をリセットすることは避け、基本的には今ある回路の問題箇所を特定して修正してください。
  - 回路全体の設計が根本的に誤っている場合など、本当にすべてをリセットする必要がある場合のみリセットしてください。
- 修正後、7.1 に戻って再試行してください。

#### 7.4. 最大試行回数に達した場合
- 5回試行してもすべてのテストが合格しなかった場合は、**作業を中断**してください。
- ユーザーに以下の内容を報告してください：
  - 回路の生成に失敗したこと
  - 各試行で発生した失敗の内容と原因の考察
  - 考えられる根本的な問題点や解決のヒント
- その後、手順9（知見まとめ）に進んでください。

### 8. 完了報告
- 全テストが合格した場合、ユーザーに成功を報告してください。
- 報告内容には以下を含めてください：
  - 成功したこと・作成した回路の概要
  - 試行した回数と、途中で修正した内容（あれば）
  - 設置した座標範囲
- その後、手順9（知見まとめ）に進んでください。

### 9. 知見のまとめ（必須）

成功・失敗に関わらず、今回の回路生成で得た知見を `knowledges/` ディレクトリにマークダウンファイルとして保存してください。

- **保存先**: `redstone_ai/knowledges/` ディレクトリ（存在しない場合は作成してください）
- **ファイル名**: 回路の種類や日時を含む分かりやすい名前にしてください（例: `and_gate_20250418.md`）
- **内容**: 以下の項目を含めてください：

```markdown
# [回路名] 生成知見まとめ

## 概要
- 目的・作成した回路の説明
- 結果（成功 / 失敗）
- 試行回数

## うまくいったこと
- 成功した設計上の工夫・アプローチ
- 有効だったブロックの組み合わせや配置

## うまくいかなかったこと
- 失敗した試みと、その具体的な症状
- テスト結果（失敗したアサーション、期待値と実際値）

## 原因の考察
- 失敗の根本的な原因と推測
- 修正によって解決した内容（成功時）
- 解決できなかった問題点と今後の課題（失敗時）

## 学んだこと・次回への改善点
- 今回の経験から得た新しい知識や気づき
- 同様の回路を作る際に役立つポイント
```
