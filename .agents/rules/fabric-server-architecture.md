# Minecraftサーバー側アーキテクチャ詳細設計

本ドキュメントでは、AIエージェント向けに提供されるMinecraft（Fabricベース）のHTTPサーバーModに関する技術的かつ詳細なアーキテクチャ設計を定義する。

## 1. 非同期アーキテクチャとスレッドセーフ設計
開発言語はJavaとし、Fabric環境を基盤とする。
AIエージェント（Python/TypeScript）からの指令はCLIツールを通じたHTTPリクエストペイロードとして送信されるが、この通信プロセスの実装において「スレッドセーフティの確保」が最も重要な課題となる。

### 1.1 ConcurrentModificationExceptionの回避とタスクスケジューリング
Minecraftのサーバーアーキテクチャは、ワールドデータの読み書き、エンティティの移動、レッドストーンの信号計算、ブロックの更新といったすべてのゲームロジックを**単一のメインスレッド（サーバーTickスレッド）**で処理する設計となっている（1 Tick = 通常50ミリ秒）。
一方、Modに組み込まれた内蔵HTTPサーバー（Netty等）は非同期のイベントループ（ワーカースレッド）で動作する。
LLMからのリクエストでワーカースレッドから直接 `world.setBlockState()` 等を呼び出すと、メインスレッドと競合し `ConcurrentModificationException` を引き起こし、即時クラッシュやチャンク破損を招く。

**【解決策: 同期と非同期のブリッジング】**
完全なスレッドセーフティを保証するため、以下の設計を厳守する。
1. HTTPリスナーでリクエストとペイロードを解析する。
2. 実際のワールド操作（取得・設置・操作）は、`MinecraftServer` インスタンス提供のメソッドを用いてメインスレッドのタスクキューにエンキュー（スケジュール）する。
3. Javaの `CompletableFuture` を利用し、メインスレッドでの実行完了をワーカースレッドが非同期に待機する。
4. 処理完了後、実行結果（成功・失敗・取得データ）をHTTPレスポンスとしてLLMに返却する。

## 2. ブロック取得・設置の高度な制御
HTTP経由でレッドストーン回路を構築する際、単なる配置以上の詳細な制御が要求される。

### 2.1 状態とNBTの取得
ブロック情報の取得APIにおいては、指定された座標の `BlockState` プロパティ（向き、動力レベル、遅延など）を正確にJSONへシリアライズする。
レッドストーンコンパレーターのモードやホッパーのインベントリ内容といった重要な情報は `BlockEntity`（旧TileEntity）に格納されているため、これらのNBTデータの情報も併せて抽出し、LLMへ提供する設計とする。

### 2.2 ブロック設置時の更新フラグ（Flags）
メソッド（`setBlockState`）に渡す「更新フラグ」の指定が極めて重要となる。
- バニラの挙動（隣接ブロックの更新、レッドストーンダストの接続変更、ピストンの反応など）を正確に再現するためには、通常**「フラグ3（ブロック状態の更新とクライアントへの同期）」**を指定する。
- 一方で、LLMが広範囲の回路ブループリントを一度に配置する場合、個別にブロック更新を行うとティックラグが発生しパフォーマンスが低下する恐れがある。
- この対策として、一時的にブロック更新を無効化する**「フラグ2」**を利用して配置処理を高速化し、後から一括でブロック更新をトリガーするといった高度な制御オプションをAPI側でサポートする。

## 3. インタラクト操作（ボタン・レバー）の完全エミュレーション
AIエージェントからの「ボタンを押す」「レバーを切り替える」といった入力をバニラの挙動に合わせて完璧にシミュレートするためには、特殊なアプローチが必要となる。

### 3.1 BlockState直接書き換えの問題点
`POWERED=true` といった形で直接BlockStateを書き換える手法には以下の致命的な欠陥がある。
1. **メカニクスの崩壊**: プレイヤーがボタンを押した際の「カチッというサウンドイベント」や「指定Tick後に自動でオフに戻るスケジューリング挙動」が実行されず、ボタンが永遠にオンのままになる。
2. **イベントのスキップ**: `PlayerInteractEvent` 等のイベントループがバイパスされるため、他のプラグイン/Modやオブザーバーの検知に意図しない副作用をもたらす。

### 3.2 FakePlayer（偽プレイヤー）パターンの実装
バニラの事象を100%完璧にシミュレートするため、実績のある「FakePlayerパターン」（Carpet ModのScarpet等でも採用）を導入する。
Fabric環境におけるブロックのアクティベーションロジックはブロッククラスの `onUse` または `use` メソッドにカプセル化されている。これを踏まえ、メインスレッド側でのインタラクト処理は以下の手順で実装する。

1. ランダムあるいは固定のUUIDとゲームプロファイル（`GameProfile`）を持つダミーの `ServerPlayerEntity`（FakePlayer）インスタンスを生成する。
2. 対象座標に基づき、クリック対象の面やヒット位置を計算して `BlockHitResult` オブジェクトを構築する。
3. 対象ブロックの `BlockState` を取得し、その `onUse` (または `use`) メソッドに対して、生成したFakePlayerと `InteractionHand.MAIN_HAND` を引数として渡して実行する。
4. 戻り値となる `ActionResult` (または `InteractionResult`) が `SUCCESS` または `CONSUME` であればインタラクト成功と見なす。

この手法により、Minecraftのコアロジックは「本物のプレイヤーが正規の手順で手動操作した」と完全に認識するため、サウンド再生、ブロック更新、Tick遅延処理などの一切のバニラメカニクスが自動的かつ正確に処理される設計となる。
