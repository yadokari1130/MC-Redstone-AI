# 4. Minecraftサーバー APIインターフェース仕様

サーバー起動時に内蔵の軽量なHTTPサーバーを立ち上げ、以下のエンドポイントを提供する。

## 4.1. `GET /api/blocks`
指定範囲のブロックリストを取得する。
- **クエリパラメータ**: `x1`, `y1`, `z1`, `x2`, `y2`, `z2` (すべて整数)。オプションで `include_entities` (ブール値) を指定可能。
- **レスポンス**:
  - `include_entities` が未指定または `false` の場合: 正常時は200 OKとブロックリストのJSON配列 `[]BlockData` を返す。配列要素にはBlockState情報に加え、対応するBlockEntityのNBTデータも含める。空気ブロック (`minecraft:air`) は原則として除外するか、フィルタオプションを設ける。
  - `include_entities` が `true` の場合: 正常時は200 OKと `{"blocks": [...], "entities": [...]}` 形式のJSONオブジェクトを返す。`entities` には範囲内のエンティティ情報（UUID、種類、座標、Yaw/Pitch、NBTデータ）が含まれる。プレイヤーエンティティは除外する。

## 4.2. `POST /api/blocks`
JSONとして送信されたブロックデータ・エンティティデータを指定座標へ一括配置する。
- **リクエストボディ**: `PlaceRequest` 形式のJSONオブジェクト。`blocks`（ブロック配列）に加え、`entities`（エンティティ配列）も同時に指定可能。高速化やバニラ挙動のエミュレートを制御するため、オプションとして「更新フラグ (Flags: 通常時は3, 高速配置時は2など)」の指定をサポートする。
- **レスポンス**: 成功した場合は200 OK。配置不能なブロックや無効な座標が含まれる場合は適宜エラーステータス（400等）を返す。

## 4.3. `POST /api/interact`
特定座標のブロックに対してインタラクト操作を実行する。
- **リクエストボディ**: `{ "x": x座標, "y": y座標, "z": z座標 }`
- **レスポンス**: インタラクト成功時は200 OK。FakePlayerパターンの実行（`onUse`）の戻り値が `SUCCESS` または `CONSUME` であれば成功とする。対象ブロックがインタラクト不能（石ブロックなど）の場合は400 Bad Request。

## 4.4. `POST /api/kill-entities`
指定範囲内のエンティティを削除する。
- **リクエストボディ**: `{ "x1": ..., "y1": ..., "z1": ..., "x2": ..., "y2": ..., "z2": ..., "type": "..." }`。`type` はオプションで、未指定時は範囲内のすべてのエンティティを対象とする。プレイヤーエンティティは削除対象から除外する。
- **レスポンス**: 正常時は200 OKと `{ "removed": 削除件数, "skipped": スキップ件数, "entities": [UUIDリスト] }` を返す。
