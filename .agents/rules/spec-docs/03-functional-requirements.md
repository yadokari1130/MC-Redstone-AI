# 3. 機能要件詳細

## 3.1. 領域ブロック情報の取得 (`get_blocks`)
指定した三次元座標（A点からB点）の範囲内に存在するブロックの情報と、各ブロックの詳細なレッドストーン状態（BlockState Properties）をJSON形式で取得する。
レッドストーンコンパレーターのモードやホッパーのインベントリ内容といった情報は `BlockEntity` に格納されているため、これらNBTデータの情報も併せて抽出してLLMに提供する。

## 3.2. ブロックの一括配置 (`place_blocks`)
AIが生成した回路構成（JSONデータ）を一括でMinecraftの世界に配置する。
対象の挙動をバニラ環境と正確に合わせるため、通常は「フラグ3（ブロック状態の更新とクライアントへの同期）」を使用する。ただし、広範囲のブループリントを一度に配置する際は、パフォーマンス低下を防ぐために「フラグ2（ブロック更新を一時無効化）」などで高速に配置し、後から一括でブロック更新を行えるような高度な制御オプションを提供する。

また、レッドストーン回路（ピストンの準接続やオブザーバーの更新検知など）では配置順序が挙動に影響を与えるため、配置を複数のフェーズに分けて順番に実行できる「フェーズ（配列）形式」をサポートする。フェーズ間には自動的に1tickの待機が入り、正しい回路動作を確保する。

## 3.3. ブロックへのインタラクト (`interact_block`)
ボタンやレバーなどの操作可能なブロックが存在する座標を指定し、人為的に右クリックした際と同様のインタラクト（状態の切り替えやパルス信号の送信）を実行する。
BlockStateの直接書き換えはバニラの挙動（サウンド再生やボタンの自動戻りなど）を毀損するため行わない。代わりに、サーバーサイドでダミーエンティティである「FakePlayer（偽プレイヤー）」を生成し、そのエンティティから対象ブロックの `onUse` または `use` メソッドを呼び出すことで、プレイヤーによる操作を完全かつ安全にエミュレートする。
